解耦原则：时间分离

❌ 错误方式

加一个「认知对抗Tab」——学生不会主动点进去。Tab切换要求主动行为，9-14岁学生的动机是做游戏，认知对抗永远不会发生。

✅ 正确方式

Gate模型——Agent在生产流水线的自然节点自动出现。Agent出现时生产层变暗但不消失（opacity 0.3），学生可以对照自己的内容回答。Agent消失后生产层完全恢复。

六个集成点：现有系统 ↔ Agent层

Agent层和现有系统的接触面只有六处。颜色标注风险等级：amber=设计变更，red=运行时风险，green=低风险加法。

R1 · AMBER
lesson.js
设计变更 每个Upgrade新增两个Agent专用字段
变更内容：agent_context（Round 1冷启动描述）和 language_dimensions[]（需要精确化的维度列表）。
风险：这两个字段的内容质量直接决定Agent追问深度。language_dimensions写得太泛（如「描述它」）→ Agent无法追到具体维度。必须在P1建立Lesson 2时逐个Upgrade仔细填写。
不影响：现有 buildPrompt() 逻辑零改动。difficulty字段保留但不驱动任何Agent逻辑。
R2 · GREEN
Upgrade.jsx
低风险加法 新增 easy_challenge_opened 事件emit
变更内容：medium_challenge_opened 和 hard_challenge_opened 已有，只需在Upgrade.jsx里对Easy Upgrade新增一行 agentBridge.trigger('easy_challenge_opened', upgradeId)。
原则：V2修正——所有Upgrade无论Easy/Medium/Hard都触发Gate 1，Upgrade难度不再决定是否追问。
R3 · GREEN
events.js
低风险加法 prompt_tab_revisited 事件已存在，AgentBridge直接监听
变更内容：events.js无需改动。AgentBridge在初始化时订阅 prompt_tab_revisited，触发Gate 2逻辑。
注意：Gate 2只在 agent_sessions 表有未验证记录时才激活。如果学生没有完成任何Gate 1，返回时不触发。
R4 · RED
agent_sessions
运行时风险 Gate 1中断导致半完整记录 → Gate 2误判
问题：Gate 1分多次写入。浏览器关闭或网络断开 → 记录存在但 upgrade_appeared IS NULL，与「完成但等待验证」的记录无法区分。Gate 2会对半完整记录发起错误验证。
修复：新增 gate1_completed boolean DEFAULT false。Gate 1最后一轮写完、continue:false返回时写入 true。Gate 2查询条件改为 gate1_completed = true AND upgrade_appeared IS NULL。
断点恢复：Gate 2同时查询 gate1_completed = false 的最近一条记录，与验证任务合并成一次对话（见Page 4详细逻辑）。
R5 · RED
sessions表
运行时风险 scheduled_end_at 为 NULL → timeRemaining = NaN → 错误模式
问题：TA未设置结束时间，或使用字段加入前创建的旧session。timeRemaining 得到NaN，NaN > 480000 返回false → 所有学生错误进入diagnose模式，即使时间充足。
修复（三层fallback）：① scheduled_end_at 有值 → 使用。② 无值 → created_at + 90分钟。③ 两者都没有 → 假设剩余20分钟（倾向retry）。
原则：宁可错误进入retry（顶多多花几分钟），不错误进入diagnose（关闭可修复的学习机会）。
TA提示：Setup.jsx在未设置结束时间时显示警告，不强制。
R6 · GREEN
reportPrompt.js
低风险加法 新增 agent_sessions 聚合section
变更内容：buildReportPrompt() 新增 agentSection，JOIN查询该学生所有 agent_sessions 记录，聚合后注入报告prompt。
边界情况：学生零条agent_sessions记录 → agentSection返回空字符串，报告正常生成，不报错。
不影响：现有V17 student_signals报告逻辑零改动。

Gate 1：完整逻辑规格

触发条件：任意Upgrade展开（easy / medium / hard，无例外）
触发方式：AgentBridge监听三个事件，统一处理
记录方式：每次触发新建一条agent_sessions记录（1:N关系）
学生展开第N个Upgrade
↓ AgentBridge新建agent_sessions记录，注入upgrade_sequence、language_dimensions_total
AgentPanel出现 · Round 1冷启动
第一句话引用 agent_context + 示范迷宫（不引用学生原话，唯一例外）
↓ 学生回答 → DeepSeek评分（具体性+因果性+自主性 = 0-9）
Round 1总分判断
≥ 6/9 → continue:false · early_release=true · actual_rounds=1 · 放行
3–5/9 → continue:true · mode:"fill" · 进入Round 2填空题
≤ 2/9 → continue:true · mode:"choice" · 进入Round 2选择题
↓ Round 2（如进入）· 必须引用Round 1学生原话
本轮比Round 1提升≥2 → continue:false · actual_rounds=2 · 放行
未提升 → continue:true · mode:"choice" · 进入Round 3
↓ Round 3（最终轮）· 无条件 continue:false · actual_rounds=3
↓ Gate 1结束 · 写入 gate1_completed=true · best_student_quote · language_growth_note
↓ AgentPanel消失 · 生产层恢复 · 学生继续复制prompt

Gate 1 AgentBridge 核心代码

// AgentBridge.js — Gate 1触发
const handleUpgradeOpened = async (upgradeId, difficulty) => {
  const upgrade = LESSON.upgrades.find(u => u.id === upgradeId)
  const existing = await supabase.from('agent_sessions')
    .select('id').eq('student_id', studentId).eq('target_upgrade_id', upgradeId)
  if (existing.data?.length > 0) return // 同一Upgrade不重复触发

  const seq = await getUpgradeSequence(studentId)
  const { data: record } = await supabase.from('agent_sessions').insert({
    student_id: studentId, session_id: sessionId,
    upgrade_sequence: seq,
    target_upgrade_id: upgrade.id,
    target_upgrade_label: upgrade.label,
    upgrade_difficulty: difficulty,
    language_dimensions_total: upgrade.language_dimensions.length,
    gate1_completed: false // R4修复：默认false，完成后写true
  }).select().single()

  agentPanel.open({
    mode: 'gate1',
    sessionRecordId: record.id,
    upgrade,
    currentPrompt: promptGenerator.getCurrentPrompt()
  })
}

三种问题模式

模式触发条件Agent行为选择题后必须
open
开放追问		 Round 1默认
Round 2且上轮≥5		 引用学生原话，追问language_dimensions中尚未覆盖的第一个维度 —
fill
填空题		 Round 1总分3-5
→ Round 2		 提供填空框架：「我要___个___，碰到___会___」，学生填入 —
choice
选择题		 Round 1总分≤2
或Round 2未提升		 提供3个选项，学生点选 必须追一句把选择变描述：「你选了[X]——[对应维度的具体追问]？」
冷启动规则（Round 1唯一例外）：Round 1第一句话引用 upgrade.agent_context + 示范迷宫，不引用学生原话。从Round 2开始，必须引用学生在上一轮说出的原话。V17「必须用学生原话提问」规则从Round 2起严格执行。

Gate 2：完整逻辑规格（含断点恢复）

触发条件：prompt_tab_revisited 事件
前置查询：AgentBridge执行两次查询，构建合并agenda
时间模式：scheduled_end_at（或fallback）计算剩余时间，>8分钟→retry，≤8分钟→diagnose

Gate 2触发：合并Agenda构建

// AgentBridge.js — Gate 2触发
const handlePromptTabRevisited = async () => {
  // 查询1：完整Gate 1，等待验证
  const { data: pendingVerify } = await supabase.from('agent_sessions')
    .select('*').eq('student_id', studentId)
    .eq('gate1_completed', true).is('upgrade_appeared', null)

  // 查询2：未完成的Gate 1（最近一条）
  const { data: incomplete } = await supabase.from('agent_sessions')
    .select('*').eq('student_id', studentId)
    .eq('gate1_completed', false)
    .order('created_at', { ascending: false }).limit(1)

  if (pendingVerify.length === 0 && incomplete.length === 0) return

  const gate2Mode = getTimeRemaining(session) > 8 * 60 * 1000 ? 'retry' : 'diagnose'

  // 合并Agenda：verify在前，resume在后
  const agenda = [
    ...pendingVerify.map(s => ({ type: 'verify', ...s })),
    ...incomplete.map(s => ({ type: 'resume', ...s }))
  ]

  agentPanel.open({ mode: 'gate2', agenda, gate2Mode })
}

// R5修复：三层时间fallback
const getTimeRemaining = (session) => {
  if (session.scheduled_end_at)
    return new Date(session.scheduled_end_at) - Date.now()
  if (session.created_at)
    return new Date(session.created_at).getTime() + 90*60*1000 - Date.now()
  return 20 * 60 * 1000 // fallback：假设20分钟，倾向retry
}

Gate 2对话流程

Gate 2开场（合并模式）
一句话提到所有待处理Upgrade，不区分verify和resume
示例：「你刚才生成了游戏——我们看看两件事：计时器出现了吗？移动陷阱我们还没聊完，一起说说。」
↓ 学生回答 → Agent评估归因质量
归因质量判断（verify部分）
完整归因（说出出现/未出现的原因）→ 直接写入，结束verify
部分归因（只说结果）→ 逐一追问每个Upgrade：「[X]出现了——prompt里写了哪句话？」
无归因（只说感受）→ 降级选择题：「这几个里，哪个你最确定是你告诉AI的？」
失败处理（upgrade_appeared = false）
追问：「你的prompt里有没有写[X]？」
A类（没写）→ 引用best_student_quote：「你说过『[quote]』，把这句话加进prompt」
B类（写了AI没做）→ 引用未覆盖维度：「你描述了[X]，但有没有说[Y]？」
时间判断
retry模式 → 给修改方向，学生重试，Gate 2重新验证，retry_count+1
diagnose模式 → 「你知道为什么没出现了吗？」记录student_diagnosed，进报告
resume部分（如有）
引用上一轮学生原话，从断点轮次继续，评分，写gate1_completed=true
Diagnose不是失败：家长报告里，diagnose模式学生显示「孩子发现了自己的prompt里缺少[X的描述]——发现问题本身是比成功更难的能力。」这是真实的认知成果。

CREATE TABLE agent_sessions (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  student_id uuid REFERENCES students(id) ON DELETE CASCADE,
  session_id uuid REFERENCES sessions(id) ON DELETE CASCADE,
  created_at timestamptz DEFAULT now(),

  -- Upgrade信息
  upgrade_sequence int NOT NULL,     -- 该学生第几个Upgrade（1/2/3...）
  target_upgrade_id text NOT NULL,
  target_upgrade_label text,
  upgrade_difficulty text,            -- 保留，不驱动逻辑
  language_dimensions_total int,
  language_dimensions_covered int DEFAULT 0,

  -- Gate 1状态
  gate1_completed boolean DEFAULT false, -- R4修复：区分完整/半完整记录
  actual_rounds int,
  early_release boolean DEFAULT false,

  -- Round 1
  round_1_input text,
  round_1_mode text,                  -- 固定"open"（冷启动）
  round_1_score_specificity int, round_1_score_causality int,
  round_1_score_autonomy int, round_1_total int,

  -- Round 2（如进入，否则NULL）
  round_2_input text, round_2_mode text,
  round_2_score_specificity int, round_2_score_causality int,
  round_2_score_autonomy int, round_2_total int,

  -- Round 3（如进入，否则NULL）
  round_3_input text, round_3_mode text,
  round_3_score_specificity int, round_3_score_causality int,
  round_3_score_autonomy int, round_3_total int,

  -- Gate 2字段（Gate 1完成后为NULL，Gate 2时填入）
  upgrade_appeared boolean,          -- NULL=未验证 true=出现 false=未出现
  student_attributed boolean,        -- 学生能说出为什么出现/未出现
  gate2_failure_type text,           -- 'no_prompt'(A类) 'prompt_ignored'(B类)
  gate2_mode text,                   -- 'retry' 'diagnose'
  gate2_input text,
  student_diagnosed boolean,         -- diagnose模式：说出了原因
  retry_count int DEFAULT 0,
  retry_appeared boolean,            -- retry后出现了吗

  -- 报告素材
  best_student_quote text,
  language_growth_note text,
  final_prompt_quality text          -- 'low' 'medium' 'high'
);

-- 索引（Gate 2查询性能）
CREATE INDEX idx_agent_sessions_student ON agent_sessions(student_id);
CREATE INDEX idx_agent_sessions_pending ON agent_sessions(student_id, gate1_completed, upgrade_appeared);

System Prompt规格

Gate 1 System Prompt

// 角色
你是认知教练，不是助手。任务是追问，不是给答案。

// 核心规则
- 不接受形容词（酷/好玩/难）→ 追问成名词或动词
- 每轮只追language_dimensions中尚未覆盖的第一个维度
- Round 1：引用agent_context+示范迷宫（冷启动唯一例外）
- Round 2起：必须引用学生上一轮原话
- 永远不替学生写prompt
- 选择题后必须追一句把选择变成描述

// 动态轮数规则
Round 1后：≥6→continue:false ｜ 3-5→continue:true,mode:"fill" ｜ ≤2→continue:true,mode:"choice"
Round 2后：提升≥2→continue:false ｜ 未提升→continue:true,mode:"choice"
Round 3后：无条件continue:false

// 返回格式（严格JSON）
{ "next_question":str, "mode":"open|fill|choice",
  "fill_template":str|null, "choices":[str]|null,
  "continue":bool,
  "scores":{"specificity":0-3,"causality":0-3,"autonomy":0-3},
  "covered_dimensions":[str],
  "best_quote":str, "language_growth":str }

// 动态注入Context
示范迷宫:{demo_maze} · Upgrade:{upgrade.label} · 描述:{upgrade.agent_context}
维度:{upgrade.language_dimensions} · 当前prompt:{current_prompt}
对话历史:{history} · 轮次:Round {n} · 上轮分:{prev_score}

Gate 2 System Prompt

// 角色
你是验证教练。帮学生发现「语言→结果」因果链，不帮学生改prompt。

// 合并模式：同时处理verify和resume
开场：一句话提到所有待处理Upgrade，不区分类型，语气连贯。
顺序：先处理verify，再处理resume。

verify：评估归因质量 → 完整/部分/无 → 升降级追问
failure：先问「prompt里有没有写[X]？」→ A类引用best_quote → B类追问未覆盖维度
时间模式 retry：给修改方向，允许重试
时间模式 diagnose：「你知道为什么没出现了吗？」记录原因，结束
resume：引用prevInput，从断点轮次继续评分

// 返回格式（严格JSON）
{ "response":str, "gate2_result":"complete|partial|none",
  "updates":[
    {"sessionId":str,"type":"verify","upgrade_appeared":bool,"attributed":bool,
     "failure_type":"no_prompt|prompt_ignored|null"},
    {"sessionId":str,"type":"resume_complete","gate1_completed":true,
     "round_N_input":str,"scores":{...},"best_quote":str}
  ],
  "continue":bool, "diagnosed_reason":str|null }

// 动态注入Context
agenda:{pending_sessions[]} · gate2_mode:"retry|diagnose"
每条session包含：upgrade_label, best_student_quote, language_dimensions, covered_dimensions, prevInput

维度库的教学价值：不同课程里，学生反复遇到相同的语言挑战（「触发条件」「结果描述」）。维度库让这些挑战跨课程保持一致的语言形式，学生在Lesson 5遇到「触发条件」时，已经在Lesson 2练习过了。跨课程的语言精确度训练形成迁移。

8 / 8 · Integration Spec

实施顺序：P1 → P2 → P3

严格按顺序执行。P1是数据和内容基础，P2是核心功能，P3是报告和监控。P2不能在P1完成前开始。

P1 — 基础层（必须全部完成才能进入P2）

P1-A
sessions表新增 scheduled_end_at
ALTER TABLE + TA Dashboard Setup.jsx时间选择器。默认值逻辑写在前端。独立可测试。
P1-B
agent_sessions表建立 + students表预留字段
完整schema含所有字段（含lesson_type · gate1_completed · 两个查询索引）。同时执行 ALTER TABLE students ADD COLUMN project_id uuid DEFAULT NULL（机器人跨课时预留，成本极低）。建完后用脚本验证INSERT/SELECT/UPDATE权限。
P1-C
Lesson 2 lesson.js — Upgrade列表 + agent字段 + 扩展端口
定义迷宫游戏所有Upgrade。同时建立三个扩展端口，以后课程零改动接入：
端口1：每个课程config加 agent: { demo_description: "..." } 块。AgentBridge读此字段，不再硬编码迷宫描述。
端口2：lesson.js顶部建立 DIMENSION_LIBRARY 常量（至少8个通用维度）。Upgrade的language_dimensions引用库，不重复写相同描述。
端口3：lesson_type字段已在P1-B的agent_sessions表里，AgentBridge建记录时写入 LESSON.id，无需额外工作。
language_dimensions的质量直接决定Agent追问的深度。「描述它如何移动」太泛，「移动方向（左右/上下/随机）」才可以被Agent引用。优先用DIMENSION_LIBRARY里的标准维度，需要课程专属描述时再自定义。
P1-D
System Prompt手动测试（不写代码）
在Claude或DeepSeek里模拟两个场景：① Easy Upgrade（计时器），1轮放行路径。② Hard Upgrade（移动陷阱），3轮降级路径。验证JSON格式稳定、评分合理、continue字段正确后，再进入P2。

P2 — 核心功能层

P2-A
AgentBridge.js
事件监听（三个upgrade_opened + prompt_tab_revisited）。Gate 1：新建记录，防重复触发。Gate 2：两次查询，合并agenda，getTimeRemaining三层fallback。独立模块，不import任何现有component。
P2-B
AgentPanel.jsx
三种mode：gate1 / gate2 / gate1_resume。动态轮数：continue字段驱动。三种问题模式：open/fill/choice渲染。生产层overlay：agentGateActive状态控制opacity和pointer-events。handleGate2Complete批量写回。
P2-C
App.jsx集成
顶层新增agentGateActive状态。AgentBridge初始化。AgentPanel挂载（z-index高层）。生产层包裹在opacity/pointer-events控制的div内。加入VITE_DEEPSEEK_API_KEY到student-app/.env.local。
P2-D
Upgrade.jsx — easy_challenge_opened
Easy Upgrade展开时新增一行agentBridge.trigger。Medium/Hard已有事件，确认emit格式一致。

P3 — 报告和监控层

P3-A
reportPrompt.js — agentSection注入
buildReportPrompt()新增JOIN查询agent_sessions。多条记录聚合。零条记录时agentSection=""，不报错。diagnose结果转化为正向报告语言。
P3-B
StudentCard.jsx — 语言精确度信号
显示early_release次数、actual_rounds分布。高优先级红色信号：3轮后upgrade_appeared=false且时间不足。
